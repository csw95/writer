# Key Items Methodology

关键物品系统用于追踪会影响剧情结果、证据链、人物选择、资源守恒或伏笔回收的物品。它解决的是“物品在哪里、谁持有、能做什么、代价是什么、何时损坏/转移/退场”的连续性问题。

## 核心定义

- **关键物品（canon/items.md）**: 记录承担剧情功能的物品画像、使用限制、持有状态和生命周期。
- **活跃关键物品（canon/items-active.md）**: 当前 Part / Volume / Arc 及后续 3-5 章需要关注的关键物品轻量索引。
- **物品真相（canon/facts.md）**: 记录物品背后的真实来源、隐藏规则、身份秘密或制度真相。
- **物品伏笔（open-loops/loops.md）**: 记录读者可见的物证、异常、缺口、误导和回收计划。

四者分工：

- `items.md` 回答“这个物品当前是什么状态、谁能用、用了会怎样”。
- `facts.md` 回答“这个物品真正是什么、背后真相是什么”。
- `loops.md` 回答“读者看见了什么问题，何时回收”。
- `state/current-state.md` 回答“本章结束时物品状态发生了什么变化”。

## 哪些物品必须建档

普通钱粮、临时衣物、一次性杂物不需要进入 `ITEM-*` 台账，只在 state 的资源变化中记录即可。

凡符合任一条件，必须先进入 `canon/items.md`，再进入章节规划：

- 会改变战斗、谈判、逃生、调查、审判、交易或阵营站位结果。
- 承担证据、信物、权柄、传承、诅咒、契约、钥匙、系统道具或禁物功能。
- 会被争夺、偷换、损毁、升级、封印、消耗、转移、继承或退场。
- 关联核心人物命运、敌对势力主动计划、关系线转折或主角阶段目标。
- 关联 `FACT-*`、`LOOP-*`，或承担物品真相、误导、反转、回收功能。
- 影响能力边界、代价体系、资源守恒或世界规则。

## 最小画像

每个关键物品至少包含：

- `ITEM-*` 稳定 ID、名称、类型。
- 当前持有者和当前位置。
- 公开认知与真实功能。
- 使用条件、代价与限制。
- 损坏、消耗、冷却、封印、丢失或转移规则。
- 来源与历史，必要时绑定 `FACT-*`。
- 关联人物、势力、地点、伏笔和章节计划。
- 首次出现、显露窗口、回收/反转/退场窗口。
- 当前生命周期阶段和禁止改写项。
- 状态变更记录。

## 生命周期阶段

关键物品必须处于以下阶段之一或组合阶段：

```text
埋设 → 初现 → 接触/获得 → 误判 → 试用 → 代价显露 → 升级/损毁/转移 → 真相显露 → 回收/退场/传承
```

阶段推进必须由 chapter_plan 声明，并在 State Manager 中更新。

## 章节规划要求

当本章涉及关键物品时，chapter_plan 必须包含 `key_item_plan`：

- 本章涉及的 `ITEM-*`。
- 当前持有者、当前位置和可用状态。
- 本章允许使用、显露、误导、转移、损毁或升级的范围。
- 使用条件、代价、冷却、资源消耗和失败后果。
- 关联 `FACT-*` / `LOOP-*` / `REL-*`。
- 对后续 3-5 章的影响。
- 禁止越界项：不得临时添加新功能，不得突然出现，不得绕过代价。

若 `items-active.md` 未覆盖本章关键物品，必须先刷新 active context，或读取 `canon/items.md` 的对应条目并同步补齐 active 文件。

## 正文执行要求

Writer 只能使用 `chapter_plan.key_item_plan` 已声明的关键物品功能。

允许：

- 写公开版本、误用、局部异常、外观变化、持有转移和代价显露。
- 通过动作、触感、重量、破损、发热、失灵、污染、耗损等可见变化呈现物品状态。
- 让角色误判物品功能，但误判必须符合公开认知和角色知情状态。

禁止：

- 用突然出现的物品解决核心矛盾。
- 让物品临时拥有未登记功能。
- 让物品无代价突破世界规则或能力边界。
- 忘记物品持有者、位置、损坏、冷却、消耗或封印状态。
- 在正文中公开 `ITEM-*` ID；正文只能写故事内名称或角色可感知描述。

## 与 Fact / Loop 的关系

- 物品的“真实身份、隐藏规则、历史来源、制度意义”属于 `FACT-*`。
- 物品的“异常表现、缺口、误导线索、未来回收问题”属于 `LOOP-*`。
- 物品的“持有者、位置、可用性、损坏/转移/退场”属于 `ITEM-*`。

一个物品可以没有核心真相，但只要有核心真相，就必须绑定 `FACT-*`。一个物品可以不承担伏笔，但只要承担长期悬念，就必须绑定 `LOOP-*` 或明确未来答案。

## 审阅与修复

Reviewer 发现以下问题时必须阻塞：

- 未建档关键物品承担核心转折、证据链、伏笔回收或危机解决功能。
- 关键物品功能、代价、持有者、位置或生命周期与 `canon/items.md` / `items-active.md` / state 不一致。
- 物品真相未登记 `FACT-*` 却承担反转、能力规则或历史因果。
- 正文通过临时物品、万能钥匙、突然失效或突然升级解决核心矛盾。

修复优先级：

1. 若只是状态遗漏，补 state / items-active。
2. 若涉及未登记关键物品，回到内容补齐，先补 `canon/items.md`。
3. 若涉及物品真相，回到事实库补齐 `FACT-*`。
4. 若涉及核心剧情解决方式，回到 chapter_plan 重写，不得硬圆。
